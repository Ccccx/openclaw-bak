> 调研范围：截止到2026年4月主流Prompt工程框架
> 
> 本文档系统梳理了16种广泛使用的Prompt框架，涵盖框架含义、范式解读及实战示例。

---

## 第一章：APE框架

### 1.1 框架概述

**APE** 全称为 **Action, Purpose, Expectation**，由研究者Mario Lyioc提出，是最早被系统化整理的Prompt框架之一。该框架通过三个核心维度来构建Prompt：**行动（Action）**定义AI需要完成的具体任务，**目的（Purpose）**阐述任务背后的动机或价值，**期望（Expectation）**描述输出结果的预期标准。

APE的核心哲学在于：从"做什么"到"为什么"再到"做成什么样"的三层递进思维。它强迫使用者先明确行动，再追问动机，最后设定验收标准，从而有效减少模糊指令。

### 1.2 对应范式

APE属于**任务导向型范式（Task-Oriented Paradigm）**，强调明确的目标和可量化的结果。该范式的核心假设是：每个Prompt都应服务于一个可完成的行动，并通过目的串联起上下文逻辑，最终由期望提供质量评判依据。

APE特别适合**单轮任务型交互**，如内容生成、数据处理、信息总结等场景。

### 1.3 示例

**示例一：内容创作场景**

```
Action（行动）：
撰写一篇关于远程办公效率提升的公众号文章，篇幅约1500字。

Purpose（目的）：
帮助企业管理者理解远程办公的利弊，并提供可落地的改进建议，
读者阅读后能够直接应用于团队管理实践。

Expectation（期望）：
文章结构清晰，分为"现状分析""核心问题""解决方案"三部分；
语言风格专业但不失亲和力；
包含至少3个可操作的实践建议；
避免空洞的理论说教，每一点建议都要有具体场景支撑。
```

**示例二：代码开发场景**

```
Action（行动）：
用Python写一个函数，实现从CSV文件中读取数据并生成统计摘要报告。

Purpose（目的）：
为数据分析团队提供一个快速了解数据集基本情况的工具，
减少人工检查数据的时间和出错率。

Expectation（期望）：
函数接受文件路径作为参数；
输出包含行数、列数、缺失值比例、数值列的均值/中位数/标准差；
代码包含清晰的注释和类型注解；
提供单元测试用例。
```

## 第二章：TAG框架

### 2.1 框架概述

**TAG** 全称为 **Task, Actor, Goal**，由Matt N（Anthropic相关社区）提出并推广。该框架以三个简洁要素构建Prompt：**任务（Task）**描述AI需要执行的动作，**角色（Actor）**定义AI扮演的身份或视角，**目标（Goal）**阐明最终要达成的成果。

TAG的设计理念是"角色赋能"——通过为AI赋予一个明确的身份（如"资深产品经理""心理咨询师"），大幅提升输出的专业性和一致性。角色的存在让AI的推理拥有了背景知识的锚点。

### 2.2 对应范式

TAG属于**角色扮演型范式（Role-Play Paradigm）**，其核心假设是：AI的行为质量高度依赖于它所扮演的角色认知。赋予合适的角色，AI就能调用该角色对应的知识体系、表达风格和决策框架。

该范式广泛用于需要领域专业知识、情感共鸣或特定视角输出的场景。

### 2.3 示例

**示例一：产品分析场景**

```
Task：对一款新上线的AI笔记应用进行竞品分析报告。

Actor：你是一位拥有10年经验的产品经理，曾在字节跳动和Notion工作，
深度了解中美 productivity 工具市场，用户研究方法论扎实。

Goal：输出一份结构化的竞品分析，包含市场份额评估、差异化亮点、
不足之处及可借鉴策略，为产品迭代方向提供决策参考。
```

**示例二：面试准备场景**

```
Task：模拟一场产品经理岗位的行为面试。

Actor：你是一位经验丰富的面试官，风格严谨专业，
擅长通过STAR法则追问细节，对产品思维、数据敏感度和沟通能力
有很高的评判标准。

Goal：通过3-5个追问，帮助候选人体会自己在项目管理中的真实能力边界，
并给出针对性的改进反馈。
```

## 第三章：CoSTAR框架

### 3.1 框架概述

**CoSTAR** 全称为 **Context, Objective, Style, Task, Audience, Response**，由SoftuiKit团队系统提出，是目前结构最完整的Prompt框架之一。六个要素层层递进：**上下文（Context）**提供背景信息，**目标（Objective）**定义核心任务，**风格（Style）**指定输出调性，**任务（Task）**描述具体动作，**受众（Audience）**明确接收对象，**响应（Response）**规定输出格式。

CoSTAR的核心哲学是"全链路设计"——从背景到格式，每个环节都有明确的指引，避免AI在关键维度上自由发挥导致偏离预期。

### 3.2 对应范式

CoSTAR属于**结构化全息范式（Structured Holographic Paradigm）**，它要求Prompt设计者对整个交互场景进行全盘思考。Context提供了信息基础，Objective指明方向，Style约束表达，Task定义动作，Audience影响内容深度，Response控制交付形式——六个要素相互制约又相互支撑。

该范式适合**复杂多轮交互**和**高质量内容交付**场景。

### 3.3 示例

**示例一：商业文案场景**

```
Context：我们公司是一款面向25-35岁职场人群的时间管理App，
近期上线了"AI日程优化"新功能，用户反馈数据良好但付费转化率偏低。

Objective：撰写一组社交媒体推广文案，提升新功能的知名度和付费转化。

Style：年轻、专业、有洞察力，像一个懂职场痛点的朋友在说话，
避免夸大其词和营销腔调，可以适度幽默。

Task：产出5条微博文案（每条不超过120字），涵盖功能介绍、
用户痛点共鸣、限时优惠三种类型。

Audience：一二线城市、注重自我提升、有付费意愿但对价格敏感的职场人。

Response：每条文案前加类型标签，用分隔线隔开，
附上每条文案适合发布的最佳时段建议。
```

**示例二：技术文档场景**

```
Context：公司内部的微服务架构正在从单体应用拆分，
新入职的后端工程师对服务间通信的协议选型缺乏了解。

Objective：编写一份内部技术培训文档，讲解gRPC与REST的选型决策指南。

Style：技术文档风格，严谨但不晦涩，假设读者有基本的HTTP和编程知识，
使用类比和图示（文字描述）来解释复杂概念。

Task：系统性地对比gRPC和REST两种协议，在性能、适用场景、
学习曲线、生态工具等维度给出分析，并给出不同场景下的选型建议。

Audience：入职1-3年的后端工程师，有Web开发经验但对现代RPC框架不熟。

Response：Markdown格式，包含对比表格、技术原理简述、
实际代码示例和常见FAQ章节。
```

## 第四章：RISE框架

### 4.1 框架概述

**RISE** 全称为 **Role, Instructions, Steps, End Goal**，由Matt N提出，是继TAG之后另一个广受欢迎的结构化框架。四个要素依次展开：**角色（Role）**赋予AI身份，**指令（Instructions）**给出总体方向，**步骤（Steps）**拆解执行路径，**最终目标（End Goal）**锁定成功标准。

RISE的核心优势在于"步骤显式化"——它明确要求将复杂任务分解为清晰的步骤序列，这不仅帮助AI更好地遵循逻辑，也便于使用者理解AI的执行思路。

### 4.2 对应范式

RISE属于**流程分解型范式（Process Decomposition Paradigm）**，该范式的核心假设是：将复杂任务显式拆解为步骤，能够显著提升AI的执行准确性和可解释性。Steps的存在让Prompt从"一次到位"变为"逐步推进"。

该范式特别适合**复杂多步骤任务**、**流程性工作**和**需要AI展示推理过程**的场景。

### 4.3 示例

**示例一：市场调研场景**

```
Role：你是一位资深市场分析师，专注于消费品行业，
擅长从公开数据中提炼商业洞察。

Instructions：帮助我完成一次针对中国即食燕麦市场的快速调研。

Steps：
1. 梳理市场规模与增长趋势（近3年数据）
2. 分析主要竞争格局（头部品牌及市场份额）
3. 识别消费者画像与核心需求
4. 总结市场机会与潜在风险
5. 给出3条可执行的进入策略建议

End Goal：输出一份10页以内的PPT大纲，每页观点明确、数据有来源、
建议有逻辑支撑，可直接用于向投资人汇报。
```

**示例二：旅行规划场景**

```
Role：你是一位私人旅行定制师，足迹遍布东南亚，
对当地文化、美食、小众景点有深入了解。

Instructions：帮一对情侣规划一趟7天的曼谷+清迈旅行。

Steps：
1. 确认旅行偏好（预算、风格、体力程度）
2. 规划每日行程路线（景点、交通、餐饮）
3. 推荐特色体验和隐藏玩法
4. 整理实用旅行贴士（签证、货币、小费文化）
5. 给出总预算估算和省钱建议

End Goal：输出一份完整的旅行手册，包含每日时间表、交通指引、
餐厅预约建议和行李清单，新婚夫妻可以直接照着走。
```

## 第五章：ICIO框架

### 5.1 框架概述

**ICIO** 全称为 **Instruction, Context, Input format, Output format**，由研究者Elvis提出，专注于Prompt的输入输出标准化。该框架四个要素：**指令（Instruction）**定义AI的核心任务，**上下文（Context）**提供必要的背景信息，**输入格式（Input format）**规定用户提供信息的方式，**输出格式（Output format）**指定AI返回结果的结构。

ICIO的设计理念是"接口标准化"——它将Prompt视为一个函数的调用，通过明确输入输出的数据格式来减少理解误差，提升AI输出的结构化程度。

### 5.2 对应范式

ICIO属于**接口契约型范式（Interface Contract Paradigm）**，该范式将AI交互类比为函数调用，强调明确的数据契约。Instruction是函数逻辑，Context是预加载参数，Input format定义参数类型，Output format定义返回值结构。

该范式特别适合**数据处理、格式转换、结构化输出**等工程化场景。

### 5.3 示例

**示例一：数据清洗场景**

```
Instruction：从以下原始数据中提取结构化信息，进行数据清洗和标准化。

Context：这是一批从不同渠道收集的客户反馈数据，
包含原始文本评论、评分、时间戳等信息，需要整合到统一格式中。

Input format：
用户提供一个JSON数组，每条记录包含：
{
  "raw_text": "...",
  "rating": "...",
  "date": "...",
  "source": "..."
}

Output format：
输出一个JSON数组，每条记录清洗后包含：
{
  "cleaned_text": "...",    // 去除emoji、特殊字符，保留核心文本
  "rating": 1-5的数字,       // 统一转换为整数
  "date": "YYYY-MM-DD",    // ISO 8601日期格式
  "source": "APP|WEB|WECHAT",  // 标准化来源渠道
  "sentiment": "positive|neutral|negative"  // 情感极性判断
}
```

**示例二：简历筛选场景**

```
Instruction：从简历文本中提取关键信息，进行结构化整理。

Context：这是一份技术岗位候选人的简历，需要提取用于 ATS（简历追踪系统）
的结构化字段，筛选用。

Input format：
用户提供简历的纯文本内容（可包含教育背景、工作经历、
技能列表、项目经验等章节）。

Output format：
输出结构化JSON：
{
  "name": "候选人姓名",
  "email": "邮箱地址",
  "phone": "电话号码",
  "education": [
    {"degree": "硕士", "major": "计算机科学", "school": "XX大学", "year": "2019"}
  ],
  "work_experience_years": 数字,
  "skills": ["Python", "SQL", ...],
  "key_projects": ["项目名称: 简短描述"],
  "summary": "3句话以内的候选人画像"
}
```

## 第六章：CRISPE框架

### 6.1 框架概述

**CRISPE** 全称为 **Capacity, Role, Insight, Statement, Personality, Experiment**，由Prompt engineering社区提出，是目前最强调"人格一致性"的框架之一。六个要素依次为：**能力（Capacity）**定义AI需要具备的知识体系，**角色（Role）**赋予AI身份，**洞察（Insight）**提供背景和专业知识，**声明（Statement）**明确任务，**人格（Personality）**设定输出风格，**实验（Experiment）**允许AI在多个方向上探索。

CRISPE的特别之处在于Experiment维度——它鼓励AI不拘泥于单一答案，而是在多个可能的方向上给出实验性的探索，这对于创意工作和分析工作都很有价值。

### 6.2 对应范式

CRISPE属于**人格模拟型范式（Persona Simulation Paradigm）**，该范式强调AI输出应深度模拟特定人格特质，而不只是在表面上使用一些词汇。Capacity和Insight提供了人格的知识基础，Personality定义了表达特征，Experiment则允许人格的多元呈现。

该范式适合**情感陪伴、高端客服、心理咨询、品牌模拟**等深度人格化交互场景。

### 6.3 示例

**示例一：心理咨询场景**

```
Capacity：你拥有临床心理学硕士学位，受过CBT（认知行为疗法）
专业训练，有5年以上青少年心理咨询经验，对抑郁和焦虑有深入研究。

Role：你是一位温暖、专业的青少年心理咨询师。

Insight：你理解当代青少年面临的压力来自学业、同伴关系、
家庭期望和社交媒体的多重夹击，你有耐心倾听而不急于给建议。

Statement：用户是一位高三学生，因为高考压力大出现了失眠和焦虑情绪，
请你以咨询师身份与其进行一场30分钟的初始访谈。

Personality：温和而坚定，不用专业术语吓人，
适度使用类比让来访者感到被理解，每次回应控制在2-3句话内，
在适当的时候用开放式问题引导。

Experiment：第一轮回应聚焦于建立信任和安全空间，
请用3种不同的开场方式来展现你的共情能力。
```

**示例二：品牌营销场景**

```
Capacity：你深度理解奢侈品行业的品牌叙事逻辑，
熟悉高净值人群的消费心理，有丰富的奢侈品文案撰写经验。

Role：你是一位奢侈品牌创意总监。

Insight：你知道真正的奢华不是炫富，而是克制和传承；
目标客群对故事和工艺的敏感度远高于对价格的敏感度。

Statement：为一款新上市的小众精粹水撰写品牌故事文案，
用于品牌官网和高端杂志内页。

Personality：克制、优雅、有历史感，像一位懂行的朋友娓娓道来，
不用夸张词汇，每句话留有呼吸感。

Experiment：请分别以"时间的艺术""匠人的手""一滴水的旅行"
三个主题方向各写一版，感受不同叙事角度的魅力。
```

## 第七章：BROOKE框架

### 7.1 框架概述

**BROOKE** 全称为 **Background, Role, Objective, Output, Process, Expectations**，由Prompt engineering社区在CO-STAR基础上演进而来。六个要素：**背景（Background）**提供项目或任务的全景信息，**角色（Role）**赋予AI专业身份，**目标（Objective）**定义核心任务，**输出（Output）**规定交付物，**流程（Process）**描述期望的执行路径，**期望（Expectations）**设定质量标准和约束。

BROOKE在CoSTAR基础上增加了"流程（Process）"维度，使得Prompt不仅定义做什么，还定义了怎么做，这大幅提升了执行路径的可控性。

### 7.2 对应范式

BROOKE属于**项目管理型范式（Project Management Paradigm）**，该范式将Prompt视为一份微型项目简报（Project Brief），包含背景、角色、目标、交付物、路径和验收标准，完整度极高，特别适合需要多方协同的复杂任务。

该范式适合**跨团队协作、方案设计、项目规划**等需要清晰流程指引的场景。

### 7.3 示例

**示例一：产品方案设计场景**

```
Background：公司计划在现有App中新增"团队共享日历"功能，
目标是在下个季度提升用户活跃度。当前DAU约50万，付费转化率3%。

Role：你是一位高级产品经理，擅长从0到1设计协作类产品。

Objective：设计"团队共享日历"功能的完整产品方案。

Output：
1. 功能优先级矩阵（Must-have / Should-have / Nice-to-have）
2. 核心功能流程图（文字描述）
3. UI原型框架描述（关键页面及交互逻辑）
4. 数据指标体系（如何衡量功能成功）
5. 技术实现风险评估

Process：
① 需求分析 → ② 竞品调研 → ③ 功能设计 → ④ 优先级排序 → ⑤ 风险评估

Expectations：
所有结论要有数据或逻辑支撑；
优先级排序需附上判断依据；
UI描述需包含主要状态（默认、空态、成功、异常）；
方案需控制在3000字以内，逻辑清晰可直接给技术团队评审。
```

**示例二：内容营销策划场景**

```
Background：我们是一家面向中小企业的B2B SaaS公司，主营CRM产品，
客单价2-5万/年，决策周期3-6周，近期需要提升MQL（营销合格线索）数量。

Role：你是一位资深的B2B内容营销策略师，服务过多家SaaS公司。

Objective：为Q3季度策划一套内容营销方案，目标MQL提升20%。

Output：
1. 内容矩阵规划（不同阶段目标受众对应的内容类型）
2. 渠道优先级建议（基于B2B决策者信息获取习惯）
3. 月度内容日历（具体选题和大致发布日期）
4. 关键内容产出模板（3种核心内容类型的框架）
5. 预算分配建议

Process：
① 目标受众分析 → ② 内容漏斗映射 → ③ 渠道选择 → ④ 内容规划 → ⑤ 资源评估

Expectations：
方案需考虑B2B决策链中的不同角色（IT/业务负责人/CFO）的关注点；
内容日历需具体到周，有可执行性；
每种内容类型需给出典型的标题和开头句示例；
总预算控制在30万以内。
```

## 第八章：ROSES框架

### 8.1 框架概述

**ROSES** 全称为 **Role, Objective, Style, Expecations, Scenario**，由Prompt engineering社区提出，专门针对场景化对话设计。五个要素：**角色（Role）**定义AI身份，**目标（Objective）**明确核心任务，**风格（Style）**设定表达调性，**期望（Expectations）**规定输出标准，**场景（Scenario）**构建具体的交互情境。

ROSES的核心创新在于"场景（Scenario）"维度——它不仅告诉AI做什么，还描述了在什么情境下做，这使得AI能够更自然地理解任务的微妙之处，如语境暗示、人际关系和情绪氛围。

### 8.2 对应范式

ROSES属于**情境嵌入型范式（Contextual Embedding Paradigm）**，该范式的核心假设是：人类交流中大量的信息通过情境隐含传递，而非显式说明。Scenario的引入让Prompt能够传递这些"隐性信息"，从而获得更符合真实场景的输出。

该范式特别适合**客户服务、销售对话、谈判模拟、培训演练**等高度情境依赖的场景。

### 8.3 示例

**示例一：销售对话场景**

```
Role：你是一位企业级软件销售顾问，在IT解决方案领域有5年经验，
擅长挖掘客户痛点并提供价值导向的解决方案。

Objective：向一位犹豫不决的潜在客户推进付费转化。

Style：专业但不高冷，理解客户的顾虑但不急于施压，
用提问引导客户自己得出结论。

Expectations：
每个回应不超过3句话；
在适当节点提出一个有力的反问；
不主动降低价格或给出未经授权的折扣；
当客户表达顾虑时，先共情再引导。

Scenario：
你正在和一位中型制造企业的IT总监通话。
他对你公司的MES（制造执行系统）解决方案很感兴趣，
但今天已经是第三次沟通了，他提到"预算可能要等到下半年"。
此刻你感受到他有所顾虑，但你不确定是价格问题、决策链问题
还是真的只是时机问题。通话已经进行了15分钟，你需要推进一步。
```

**示例二：绩效反馈场景**

```
Role：你是一位技术团队Leader，正在进行季度绩效面谈。

Objective：给一位表现中等偏上、但最近两个项目都略有延期的工程师
提供建设性的反馈和发展建议。

Style：直接但不伤人，认可贡献的同时诚实指出改进点，
以成长为导向，给出具体可操作的建议。

Expectations：
反馈要有具体事例支撑，避免空泛评价；
优点和不足的比例约6:4；
每个改进建议都要有具体的行动步骤；
最后给出一个90天的个人发展目标。

Scenario：
小李是你团队的后端工程师，工龄3年，代码质量一直不错，
同事关系也好。但Q4的项目延期了两周，Q1的API重构也超期了5天。
你分析主要原因是他在复杂任务上没有提前拆分和排期，
总是等项目快到期了才发现风险。你认为他的能力上限远不止于此，
但需要改进时间管理方式。今天的绩效面谈大约30分钟。
```

## 第九章：SCOPE框架

### 9.1 框架概述

**SCOPE** 全称为 **Scenario, Context, Objective, Planning, Evaluation**，由Prompt engineering社区提出，专门为复杂推理和规划任务设计。五个要素：**场景（Scenario）**描述当前状态或问题背景，**上下文（Context）**提供必要的背景知识，**目标（Objective）**定义成功标准，**规划（Planning）**要求AI生成具体行动计划，**评估（Evaluation）**定义评判计划质量的标准。

SCOPE的核心创新是"规划（Planning）"和"评估（Evaluation）"的结合——它不仅让AI做计划，还让AI先审视自己的计划，从而提升输出的质量和可靠性。

### 9.2 对应范式

SCOPE属于**元规划型范式（Meta-Planning Paradigm）**，该范式将AI的推理过程本身纳入Prompt设计。Planning要求AI展示推理路径，Evaluation则提供了自我修正的机制，形成了一个"计划→评估→改进"的内部循环。

该范式特别适合**战略规划、风险评估、复杂决策、多方案比较**等需要深度思考的场景。

### 9.3 示例

**示例一：创业战略规划场景**

```
Scenario：公司成立18个月，做AI简历优化工具，
ARR约80万，月均新增付费用户约200人，团队12人，
近期刚完成A轮融资。

Context：AI应用赛道竞争激烈，头部产品已占据先发优势，
资金相对充裕但不能长期烧钱。团队技术能力强但缺乏商业化经验。
监管政策存在不确定性（AIGC相关法规）。

Objective：制定未来12个月的产品和商业化战略。

Planning：
请按以下步骤展开：
① 内外部分析（SWOT或类似框架）
② 确定3个关键战略方向
③ 为每个方向设计具体的执行路径（6-3-3原则：6个月目标/3个月里程碑/3个月快速验证）
④ 识别关键风险及应对预案

Evaluation：
请用以下标准自评你的方案：
- 可执行性：计划是否有明确的行动步骤，而非空洞方向？
- 可验证性：每个里程碑是否有可量化的指标？
- 资源匹配：是否考虑了当前团队规模和资金状况？
- 风险意识：是否覆盖了主要已知风险并有应对方案？
请在方案末尾附上自评结果，并说明方案的最大优势和最大风险。
```

**示例二：技术选型决策场景**

```
Scenario：团队需要为一个日均PV约5000万的数据平台选型新的消息队列系统，
现有系统 Kafka 运维成本过高，需要评估替代方案。

Context：团队规模15人，有Python和Java技术栈，
DevOps能力中等，目前使用阿里云ECS自建。
对消息可靠性要求高（金融级数据），延迟敏感（需要<100ms P99）。

Objective：做出一个充分论证的技术选型建议。

Planning：
① 收集候选方案（至少3个：Kafka / RocketMQ / RabbitMQ / Pulsar）
② 制定评估维度（性能、可靠性、运维成本、扩展性、生态、社区）
③ 进行多维度对比分析
④ 结合团队实际情况给出推荐
⑤ 给出分阶段迁移策略

Evaluation：
- 每个维度是否有具体数据支撑（benchmark或公开报告）？
- 推荐方案是否充分考虑了团队现状的约束条件？
- 迁移策略是否包含回滚方案？
- 是否识别了选型后3个月内最可能遇到的问题？
```

## 第十章：CREATE框架

### 10.1 框架概述

**CREATE** 全称为 **Character, Request, Examples, Adjust, Type, Extras**，由Prompt engineering社区提出，专门针对**创意写作和内容创作**场景优化。五个核心要素：**角色/人物（Character）**定义叙事视角或主角，**请求（Request）**明确创作需求，**示例（Examples）**提供参考风格或结构，**调整（Adjust）**说明偏好和限制，**类型（Type）**指定内容形式，**额外要求（Extras）**补充特殊需求。

CREATE的独特之处在于"示例（Examples）"维度——它明确鼓励通过少量示例（Few-Shot）来引导AI理解期望的写作风格，这比纯文字描述要高效得多。

### 10.2 对应范式

CREATE属于**创意引导型范式（Creative Direction Paradigm）**，该范式的核心假设是：创意写作中，最有效的引导方式不是描述规则，而是展示"对的作品长什么样"。Examples提供了这个参照系，让AI的创意输出在正确的方向上展开。

该范式专用于**小说创作、剧本编写、营销文案、品牌故事**等高度依赖风格和调性的创意场景。

### 10.3 示例

**示例一：短篇小说创作场景**

```
Character：主角是一位82岁的退休钢琴教师，在老城区的
一栋旧公寓里独居，她已经5年没有弹过钢琴了。
她的女儿在另一个城市，每年只回来两次。

Request：写一个2000字左右的短篇小说，讲述一个下午发生的故事。

Examples（参考风格）：
参考村上的短篇叙事节奏——克制的情感、细腻的日常细节、
出人意料但合乎情理的结尾。开头示例：
"出租车拐进那条窄巷的时候，雨刚好停了。
玛丽亚从车窗望出去，看见公寓楼三层的窗帘被风吹起一角。
那是她小时候常常看着母亲拉开的那扇窗。"
——请保持这种平静表面下暗流涌动的叙事张力。

Adjust：
- 语调：平静、内敛，避免过度煽情
- 节奏：缓慢，有留白
- 视角：第三人称有限视角，聚焦于老人
- 不要给出确定性的结局，留有余韵

Type：文学短篇小说

Extras：
请在故事末尾留一段30字以内的"点睛之笔"（单独成段），
作为整篇的情感落点。
```

**示例二：品牌故事创作场景**

```
Character：品牌主角是一个手工皮具工坊，
由一位从金融行业转行的80后创立，
工坊位于大理，用当地皮革和传统手艺制作皮具。

Request：写一篇800字的品牌故事。

Examples（参考风格）：
参考"例外"和"无用"等国内小众品牌的叙事调性——
真实、克制、有温度，不煽情但让人动容。
开头示例：
"2015年，David辞掉了陆家嘴的工作，
在大理古城外租下了一间带院子的老房子。
他不知道自己要做什么，只知道自己不想再做什么。"
——请保持这种真实的个人叙事感。

Adjust：
- 不美化创业艰辛，但展现真实的热爱
- 融入大理的地域特质和手工艺的工匠精神
- 结尾要有品牌的核心理念升华
- 避免"匠心""初心"等被用烂的词汇

Type：品牌故事长文案

Extras：
提供3个备选标题，每个标题不超过15个字，
并说明每个标题的适用场景。
```

## 第十一章：ERA框架

### 11.1 框架概述

**ERA** 全称为 **Experience, Role, Action**，由Prompt engineering社区在2024年提出，是一种极简但高效的Prompt框架。三个要素：**经验（Experience）**赋予AI相关领域的经验背景，**角色（Role）**定义AI的身份，**行动（Action）**明确需要完成的具体动作。

ERA的设计哲学是"少即是多"——它只保留了最核心的三个维度，刻意去除冗余结构，让Prompt更加精炼。经验（Experience）这个维度在其他框架中很少被单独强调，但实际效果证明它对提升AI输出的专业深度非常重要。

### 11.2 对应范式

ERA属于**经验锚定型范式（Experience Anchoring Paradigm）**，该范式的核心假设是：AI输出的质量高度依赖于它被"赋予"了多少可信的经验背景。Experience维度通过赋予AI虚构但具体的经验，让AI的推理拥有了"肌肉记忆"式的专业直觉。

该范式适合**需要快速上手、简洁高效**的日常交互场景，以及**专家级输出**场景。

### 11.3 示例

**示例一：法律咨询场景**

```
Experience：你是一位处理劳动纠纷超过15年的律师，
代理过上百起劳动仲裁案件，熟悉劳动合同法、司法解释
和各地仲裁委的裁判惯例，尤其擅长处理互联网公司的裁员纠纷。

Role：专业劳动法律顾问

Action：评估以下情况中劳动者的权益边界：
一位在滴滴工作4年的司机（合同签的是"合作框架协议"而非劳动合同），
平台以"连续评分低于4.6"为由封禁了其账号。
请从法律关系定性、权益主张路径、胜诉可能性三个维度给出分析。
```

**示例二：装修设计场景**

```
Experience：你是一位从业12年的室内设计师，专注于
中小户型（50-90平）的空间优化，擅长北欧和日式简约风格，
对收纳系统和动线规划有深入研究，自己也装修过5套房子，
踩过各种坑。

Role：私人装修顾问

Action：帮分析一套60平两室一厅的户型改造方案。
承重墙位置已标注，层高2.7米，采光主要来自南向客厅。
常住人口：夫妻二人 + 1岁幼儿 + 一位帮忙带孩子的老人。
核心诉求：增加储物空间，老人房需要安静安全，客厅需要
同时满足大人办公和幼儿活动。
请给出3个不同预算等级（简装/标准/精装）的改造方向建议。
```

## 第十二章：DECR框架

### 12.1 框架概述

**DECR** 全称为 **Define, Example, Constraint, Refine**，由Prompt engineering社区在2025年提出，专门针对**迭代优化型任务**设计。四个要素：**定义（Define）**明确任务目标和成功标准，**示例（Example）**提供期望输出的参照，**约束（Constraint）**设定限制条件和边界，**精炼（Refine）**要求AI主动识别模糊点并提出优化建议。

DECR的核心创新是"精炼（Refine）"维度——它要求AI不仅执行任务，还要主动识别Prompt本身的模糊之处，并提出澄清问题，从而提升最终输出的质量。这个框架特别适合与AI进行多轮对话迭代。

### 12.2 对应范式

DECR属于**迭代优化型范式（Iterative Refinement Paradigm）**，该范式的核心假设是：好的Prompt不是一次写出来的，而是在AI的反馈下逐步完善的。Refine维度引入了一个"元认知"环节，让AI扮演Prompt优化顾问的角色。

该范式特别适合**需求不明确、探索性任务、多轮对话优化**场景。

### 12.3 示例

**示例一：演讲稿撰写场景**

```
Define：
为一个科技公司的CEO撰写年会演讲稿。
目标：激励团队、增强凝聚力、传递公司战略方向。
时长：约20分钟（约4000字）。
背景：过去一年公司经历了战略转型，营收增长30%但团队也有20%的裁员。

Example：
参考Tim Cook在苹果发布会上的叙事风格——
自信但不傲慢，数据与故事交织，结尾有情感力量。
开头示例：
"一年前，有人问我：苹果还值得相信吗？
今天我想用一个词来回答——创新。
不是我们挂在墙上的词，而是我们每天在做的事。"
——请保持这种有力量感但接地气的风格。

Constraint：
- 总字数不超过4200字
- 包含至少3个具体的团队/个人故事
- 提及数字时要具体（不能只说"大幅增长"）
- 避免过度煽情和空洞的口号
- 演讲稿需要区分"正式段落"和"可脱稿发挥"的即兴段落

Refine：
在正式输出演讲稿之前，请先列出你对这次任务的3个理解疑点
（例如：目标受众的年龄层、往年年会的风格基调、"裁员"话题的处理尺度等），
并给出你的最佳假设，以便我在你开始撰写前进行确认或修正。
```

**示例二：App功能设计场景**

```
Define：
为一款面向老年人的健康管理App设计"用药提醒"功能模块。
目标：帮助65岁以上老年慢病患者规律服药，降低漏服和误服风险。

Example：
参考"美篇"和"支付宝大字版"等适老化产品的设计思路——
超大字体、高对比度、单任务聚焦、操作步骤最少化。
某适老化App的服药记录页面设计示例：
"① 8:00 阿司匹林 1片 ✓ 已服用
② 12:00 降压药 1片 ○ 待服用
——整页只显示今日待办，没有其他干扰信息。"

Constraint：
- 必须考虑视力下降、手指不灵活、认知衰退等老年特点
- 不能依赖复杂的操作流程（最多2步完成操作）
- 需要考虑子女远程协助查看的需求
- 符合国家药监局关于用药提醒的相关规范
- 技术实现基于现有Android/iOS双平台

Refine：
在开始详细设计之前，请先提出你对老年用户用药场景的
5个关键假设（例如：老人是否能够自主设置提醒？
子女在用药管理中应该扮演什么角色？
忘记服药时应该用什么方式提醒最有效？），
我确认后再开始详细设计。
```

## 第十三章：HEART框架

### 13.1 框架概述

**HEART** 全称为 **History, Expectations, Assets, Rules, Tone**，由Prompt engineering社区在2025年提出，专为**长程对话和项目推进**场景设计。五个要素：**历史（History）**记录之前对话或任务的上下文，**期望（Expectations）**明确目标与预期，**资产（Assets）**提供可用的参考资料，**规则（Rules）**设定约束和行为边界，**语调（Tone）**定义交互风格。

HEART的核心价值在于"上下文管理"——它将之前分散在不同位置的上下文信息（历史、资产、规则）统一到一个框架中，特别适合需要多轮迭代的复杂项目。

### 13.2 对应范式

HEART属于**项目管理+上下文连续性范式（Project Management + Context Continuity Paradigm）**，该范式的核心假设是：复杂任务的Prompt质量取决于上下文的完整性和一致性。History确保连续性，Expectations确保方向，Assets提供资源，Rules设置边界，Tone确保风格一致。

该范式适合**长程项目、文档撰写、产品开发、咨询服务**等多轮交互场景。

### 13.3 示例

**示例一：产品需求文档（PRD）撰写场景**

```
History：
我们已完成第3轮用户访谈（共12位目标用户），
核心痛点集中在：无法快速找到之前保存的内容（8人提及）、
与家人共享信息不方便（7人提及）、界面层级太深（5人提及）。
第2轮访谈后我们否定了"社交化"方向，认为不符合隐私诉求。

Expectations：
输出完整的PRD文档，包括功能描述、优先级、原型框架、
数据指标和上线计划。文档需要通过技术团队评审，
可以直接作为开发依据。

Assets：
- 用户访谈原始记录（已整理成摘要）
- 竞品分析报告（已完成）
- 现有App的信息架构图（附件中）
- 技术约束文档（来自研发团队）

Rules：
- 所有功能描述必须对应到具体的用户痛点（有访谈依据）
- 不做竞品没有验证过的创新功能
- 优先级必须用MoSCoW框架标注
- 每个功能需要有验收标准（可测试）

Tone：
专业、高效、数据驱动，文档结构严谨，
段落开头用结论先行，每页有摘要。
避免"为了提升用户体验"这种模糊表述。
```

**示例二：年度营销方案制定场景**

```
History：
去年Q4我们做过一次私域流量测试，3个500人社群运营1个月，
月活约15%，转化率2.3%。主要问题是内容同质化和缺乏个性化触达。
今年预算同比减少20%，团队从5人缩减到3人。

Expectations：
制定2026年全年私域运营方案，在预算削减的情况下
力争私域用户增长30%、月活提升至25%、付费转化提升至4%。

Assets：
- 去年Q4私域测试的完整数据报表
- 现有用户画像分析报告
- 竞品私域运营策略调研
- 可用的自动化营销工具清单

Rules：
- 预算严格控制在去年的80%以内
- 3人团队配置，每人最多负责1个核心指标
- 所有方案必须可以在一周内启动MVP验证
- 避免过度依赖外部服务商（自持优先）

Tone：
务实、可执行、有数据敏感度，
方案需体现"少即是多"的运营哲学，
每个季度有明确的复盘和调整节点。
```

## 第十四章：TRACE框架

### 14.1 框架概述

**TRACE** 全称为 **Task, Resources, Actors, Criteria, Execution**，由Prompt engineering社区在2025年提出，专注于**团队协作和流程设计**场景。五个要素：**任务（Task）**定义核心工作项，**资源（Resources）**列出可用资产，**参与者（Actors）**定义角色及职责，**标准（Criteria）**设定验收条件，**执行（Execution）**描述实施步骤。

TRACE的核心优势是"角色-资源-标准"的三维对齐——它不仅告诉AI要做什么，还明确分配了谁用什么资源做什么，这使得输出可以直接用于团队分工和项目管理。

### 14.2 对应范式

TRACE属于**团队协同型范式（Team Collaboration Paradigm）**，该范式的核心假设是：复杂任务的有效执行需要明确"谁用什么做，做到什么程度"。TRACE通过Actors和Resources的显式定义，解决了多角色协作中的责任模糊问题。

该范式适合**跨职能团队协作、项目启动会、流程设计、SOP制定**等需要明确分工的场景。

### 14.3 示例

**示例一：电商大促活动策划场景**

```
Task：策划并执行"618"电商大促活动，目标是GMV同比增长25%。

Resources：
- 活动预算：80万（其中流量采购50万、补贴20万、内容制作10万）
- 可用渠道：App弹窗、公众号、抖音账号（50万粉丝）、短信
- 数据工具：生意参谋、京东商智、自有CDP用户数据平台
- 外部资源：3个MCN机构、2个KOL合作档期

Actors：
- 产品经理：负责活动机制设计（优惠券、满减、抽奖规则）
- 运营：负责活动页面搭建和日常运营监控
- 市场：负责站外推广和KOL对接
- 客服：负责活动期间的客诉处理和FAQ准备
- 数据分析师：负责实时数据监控和活动复盘

Criteria：
- 活动UV ≥ 500万
- 转化率 ≥ 3.5%
- 客诉率 < 0.5%
- 活动ROI ≥ 1:4
- 页面加载时间 < 3秒

Execution：
① 活动机制确定（T+0）→ ② 页面设计评审（T+3）→ ③ 技术开发（T+10）
→ ④ 内测与压力测试（T+14）→ ⑤ 预热期（T+15至T+17）
→ ⑥ 正式活动（T+18至T+20）→ ⑦ 收尾与复盘（T+21至T+25）
```

**示例二：新功能上线发布流程场景**

```
Task：为新上线的"智能推荐"功能制定完整的上线发布计划。

Resources：
- 测试环境：staging环境（已通过功能测试和性能测试）
- 生产环境：主站（DAU 100万）、App（iOS/Android）
- 监控工具：Prometheus、Grafana、自建报警系统
- 回滚方案：特性开关（Feature Flag）支持秒级回滚

Actors：
- 后端工程师：负责API和数据服务的发布
- 前端工程师：负责推荐模块UI的更新
- 测试工程师：负责UAT测试和回归测试
- 运维/DevOps：负责发布流程编排和生产环境监控
- 产品经理：负责验收和灰度策略制定

Criteria：
- 核心接口P99响应时间 < 200ms
- 推荐准确率（A/B测试）相比基线提升10%以上
- 无P0/P1级Bug
- 灰度期间Crash率不高于基线

Execution：
① 灰度5%用户（T+0）→ ② 数据监控48小时（T+2）→ ③ 灰度20%（T+5）
→ ④ 全量上线（T+8）→ ⑤ 发布后48小时重点监控（T+10）→ ⑥ 正式复盘（T+14）
```

## 第十五章：REFINE框架

### 15.1 框架概述

**REFINE** 全称为 **Review context, Clarify objective, Format output, Include examples, Navigate constraints, Elevate quality**，由Prompt engineering社区在2025年提出，是一种六阶段的迭代式Prompt框架。六个步骤依次展开：**回顾上下文（Review context）**→ **明确目标（Clarify objective）**→ **格式化输出（Format output）**→ **包含示例（Include examples）**→ **导航约束（Navigate constraints）**→ **提升质量（Elevate quality）**。

REFINE的核心哲学是"质量内建于流程"——每个维度都在Prompt中设置了质量把关点，形成了一个六层的质量过滤网。特别适合对输出质量要求极高的专业场景。

### 15.2 对应范式

REFINE属于**质量内建型范式（Quality Built-In Paradigm）**，该范式的核心假设是：Prompt质量决定了输出质量的上限，而质量的保障需要通过系统化的设计来实现，而非依赖AI的自我改进。六个维度环环相扣，每一环都在提升输出的可靠性。

该范式适合**专业报告、技术方案、学术写作、合规审查**等高标准场景。

### 15.3 示例

**示例一：技术架构评审场景**

```
Review context：
我们正在将一个单体电商应用拆分为微服务架构，
当前团队12人，计划分3个阶段迁移。
已完成第一阶段（用户服务和订单服务），正在规划第二阶段。

Clarify objective：
输出一份第二阶段微服务拆分的技术架构方案，
包括要拆分的边界、数据迁移策略、测试策略和风险预案。
方案需要通过CTO和技术评审委员会的评审。

Format output：
- 执行摘要（1页）
- 当前架构问题分析
- 目标架构设计（含架构图ASCII描述）
- 详细实施计划（含甘特图文字版）
- 数据迁移方案
- 测试策略
- 风险评估矩阵（概率×影响）
- 团队能力和资源需求

Include examples：
参考Uber和Netflix的微服务迁移经验，
每个关键决策点附上业界实践案例。
格式示例：
"【决策点：是否使用事件溯源】
Uber在支付模块采用了事件溯源，显著提升了审计能力，
但复杂度增加约30%。我们建议..."
```

Navigate constraints：
- 必须在现有Kubernetes集群上运行，不引入新的基础设施
- 迁移期间必须保证零停机
- 单个微服务部署包不超过500MB
- 团队对新框架的学习曲线不超过2周

Elevate quality：
- 所有技术选型必须有具体的性能/成本数据支撑
- 实施计划中每个里程碑需有可量化的验收标准
- 风险预案需覆盖"最坏情况"的应对方案
- 方案需包含"快速失败"机制——如果拆分遇到不可逾越的障碍，
  明确的放弃标准和回退路径是什么？
```

**示例二：市场研究报告场景**

```
Review context：
公司正在考虑进入东南亚市场（印尼、越南、泰国），
需要评估三个市场的机会和风险。
已有的信息：竞品已先行进入，当地合作伙伴关系待建立。

Clarify objective：
输出一份三个市场的综合评估报告，
包含市场规模、竞争格局、进入壁垒、监管环境、合作伙伴建议，
最终给出优先级排序和进入策略建议。
报告需要用于董事会决策。

Format output：
- 执行摘要（1页，结论先行）
- 市场概览（每个市场独立章节）
- 竞争分析（三个市场横向对比）
- 监管环境（每个市场）
- 合作模式建议
- 风险评估
- 进入策略建议（含优先级和时间表）
- 附件：数据来源说明

Include examples：
参考以下已公开的出海案例：
- 字节跳动TikTok东南亚扩张路径（快速本地化策略）
- 阿里巴巴Lazada的收并购整合路径（平台策略）
- 传音控股非洲市场策略（性价比+本地化）
每个案例提取1-2个可借鉴的关键洞察。

Navigate constraints：
- 数据来源必须为公开信息或一手调研，避免使用未经核实的数据
- 每个市场至少覆盖2个一手访谈信息源
- 报告篇幅控制在40页以内（不含附件）
- 规避敏感话题（政治、宗教、领土争议相关内容不纳入）

Elevate quality：
- 给出明确的市场优先级建议（而不是模糊的"三个市场都有机会"）
- 每个结论都需要有数据或逻辑链条支撑
- 识别3个最关键的不确定性因素，并给出"如果...那么..."的应对场景
- 在报告末尾提供3个不同风险偏好等级的进入策略版本
（激进/稳健/保守），供决策者选择
```

## 第十六章：SAME框架

### 16.1 框架概述

**SAME** 全称为 **Situation, Analysis, Mitigation, Execution**，由Prompt engineering社区在2025年提出，专门针对**问题诊断和解决**场景设计。四个步骤：**情况（Sribution）**描述当前状态和问题，**分析（Analysis）**深入挖掘根因，**缓解（Mitigation）**制定应对措施，**执行（Execution）**给出实施路径。

SAME的设计受到了医学诊断思维的启发——先描述症状（S），再分析病因（A），然后制定治疗方案（M），最后确定治疗步骤（E）。这是一个闭环的结构，适合处理各类问题诊断和应急响应场景。

### 16.2 对应范式

SAME属于**诊断-解决闭环型范式（Diagnosis-Solution Loop Paradigm）**，该范式的核心假设是：好的问题解决不是"头痛医头"，而是系统性地经历"症状描述→根因分析→方案制定→路径执行"的完整闭环。SAME的每一步都建立在前一步的基础上，形成不可跳过的逻辑链条。

该范式特别适合**故障诊断、应急响应、问题复盘、战略调整**等需要系统性思考的场景。

### 16.3 示例

**示例一：业务指标下滑诊断场景**

```
Situation：
产品DAU从上周三开始持续下滑，从日均80万跌至今日的65万，
降幅约19%。同时，新增用户数量与上周基本持平，
但次日留存率从42%下降到31%，7日留存从28%下降到19%。
技术侧未收到异常报警，系统性能指标正常。

Analysis：
请从以下维度进行根因分析：
① 用户行为维度：用户使用时长、核心功能使用频率、流失节点
② 外部因素维度：竞品动态、App Store评分和评论变化、社会事件
③ 产品维度：近期是否发版、是否修改了核心流程、推送策略是否变化
④ 渠道维度：各获取渠道的留存是否一致
请给出最可能的3个根因假设，并附上每个假设的支持证据和反驳证据。

Mitigation：
针对每个根因假设，制定对应的短期应急措施（1周内可执行）
和中长期解决方案（1个月以上）。
标注每个措施的资源需求和预计效果。

Execution：
按优先级排列所有措施，制定未来2周的执行计划，
包含每日行动项、负责人和验收标准。
识别执行过程中最大的风险点及预案。
```

**示例二：技术故障复盘场景**

```
Situation：
上周五 14:23 至 16:45，产品核心功能不可用约2小时12分钟。
故障原因是数据库主从切换时，从库复制延迟导致查询了大量过期数据，
触发了一个未做防护的缓存穿透，最终导致服务雪崩。
影响范围：约12万活跃用户，期间所有写操作失败，读操作成功率约40%。
技术团队在15:10定位到根因，15:35完成紧急修复，16:45观察稳定后关闭报警。

Analysis：
请从以下维度进行深度复盘：
① 技术根因：从库延迟→缓存穿透→雪崩的完整链路分析
② 流程根因：代码发布流程、测试覆盖情况、监控报警机制是否有效
③ 团队根因：人员配置、知识传递、on-call轮值是否合理
④ 架构根因：系统哪些地方缺乏保护机制（熔断、降级、限流）
请给出每个维度的根因结论，用"5-Why"方法追溯到最底层原因。

Mitigation：
针对每个根因，制定对应的技术改进措施（短期/中期/长期）。
技术改进需有具体的方案描述和预期效果。

Execution：
制定未来1个月的改进计划，按优先级排序，
每个改进项需有明确的执行人、验收标准和完成时间。
特别关注那些"看起来简单但容易忽略"的基础改进项。
最后，提供一份给工程团队的故障复盘报告模板，
确保下次同类故障能在30分钟内完成定位和修复。
```
