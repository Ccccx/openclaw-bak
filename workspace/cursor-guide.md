## 1. 入门基础

### 1.1 Cursor 是什么？

Cursor 是一个 AI 代码编辑器和编码 Agent，基于 VS Code 构建。它代表了软件开发的未来方式：

```
┌─────────────────────────────────────────────────────────────┐
│                    Cursor 架构                              │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────┐    ┌──────────┐    ┌──────────┐            │
│  │   Tab    │    │  Cmd+K   │    │  Agent   │            │
│  │  代码补全 │    │  智能编辑 │    │  自主编码 │            │
│  └──────────┘    └──────────┘    └──────────┘            │
├─────────────────────────────────────────────────────────────┤
│                    AI Model Layer                          │
│         (OpenAI / Anthropic / Gemini / xAI)               │
├─────────────────────────────────────────────────────────────┤
│                    Harness Layer                           │
│      指令调优 + 工具编排 + 上下文管理                       │
├─────────────────────────────────────────────────────────────┤
│                    VS Code Core                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心快捷键

| 功能 | 快捷键 | 说明 |
|------|--------|------|
| **Tab** | `Tab` | 接受 AI 建议的代码补全 |
| **Cmd+K** | `⌘+K` (Mac) / `Ctrl+K` (Windows) | 打开 AI 编辑面板 |
| **Cmd+L** | `⌘+L` | 打开 AI 对话框 (Agent) |
| **Cmd+/** | `⌘+/` | 添加注释/取消注释 |
| `Cmd+Shift+R` | `⌘+Shift+R` | 重构选中代码 |
| `Cmd+Shift+K` | `⌘+Shift+K` | 删除整行 |

### 1.3 快速开始

```bash
# 1. 下载 Cursor
# 访问 https://cursor.com/download

# 2. 打开项目
cursor .

# 3. 基本使用流程
# - Cmd+K: 选择代码 → 输入修改指令 → 预览 → 接受
# - Cmd+L: 输入复杂任务 → Agent 自动完成
```

## 2. 核心编辑功能

### 2.1 Tab - 智能代码补全

**工作原理：**
Cursor 的 Tab 不是传统意义上的自动补全，而是**预测你接下来要写的代码**。

```python
# 示例：当你输入
def calculate_sum(a, b):
    return a + b

# Cursor 可能直接显示完整的后续代码：
def calculate_average(a, b, c):
    return (a + b + c) / 3
```

**最佳实践：**
- Tab 适合**熟悉的代码模式**
- 按 `Tab` 接受，按 `Esc` 拒绝
- 不要过度依赖，保持对代码的理解

### 2.2 Cmd+K - 精准代码编辑

**使用场景：** 单文件修改、重构、生成片段

```
┌────────────────────────────────────────────┐
│  Cmd+K 编辑流程                            │
├────────────────────────────────────────────┤
│                                            │
│  1. 选中代码 / 选中无 = 整文件上下文        │
│       ↓                                    │
│  2. 输入修改指令                           │
│       ↓                                    │
│  3. AI 生成修改预览                        │
│       ↓                                    │
│  4. 审查差异 (Diff View)                   │
│       ↓                                    │
│  5. 接受 (Cmd+Shift+Enter) / 拒绝 (Esc)    │
│                                            │
└────────────────────────────────────────────┘
```

**示例指令：**
```
# 重构
"将这个函数改写成箭头函数"

# 添加功能
"为这个组件添加 TypeScript 类型"

# 修复 bug
"修复这个排序函数的边界情况"

# 生成测试
"为这个函数生成单元测试"
```

### 2.3 Cmd+L - Agent 对话模式

**工作原理：**
Agent 是 Cursor 最强大的功能，可以自主完成复杂的多文件任务。

```
┌────────────────────────────────────────────────────────────┐
│                    Agent 能力                              │
├────────────────────────────────────────────────────────────┤
│ ✓  理解整个代码库语义                                      │
│ ✓  搜索文件和代码                                          │
│ ✓  编写和编辑多个文件                                      │
│ ✓  执行终端命令                                            │
│ ✓  运行测试                                                │
│ ✓  创建 Git 提交和 PR                                      │
│ ✓  控制浏览器进行 UI 验证                                  │
└────────────────────────────────────────────────────────────┘
```

**模式切换：**
- **Normal Mode**: 直接执行任务
- **Plan Mode** (`Shift+Tab`): 先规划，再执行

## 3. Agent 模式

### 3.1 Plan Mode（计划模式）

**为什么要用计划模式？**

> 芝加哥大学的研究表明：经验丰富的开发者更倾向于在生成代码之前进行规划。

```
┌─────────────────────────────────────────────────────────────┐
│                   Plan Mode 工作流                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   输入任务                                                  │
│      ↓                                                     │
│   ┌─────────────────────────────────────┐                  │
│   │  1. 研究代码库找相关文件              │                  │
│   │  2. 询问澄清问题                     │                  │
│   │  3. 创建详细实现计划 (Markdown)       │                  │
│   │  4. 等待用户审批                     │                  │
│   └─────────────────────────────────────┘                  │
│      ↓                                                     │
│   用户审查/编辑计划                                         │
│      ↓                                                     │
│   "开始构建" → Agent 执行                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**使用技巧：**
- 点击 **"Save to workspace"** 保存计划到 `.cursor/plans/`
- 直接编辑 Markdown 文件调整步骤
- 计划文件可以团队共享

### 3.2 长期运行 Agent (Long-running Agents)

**最新功能：** 2026年2月推出的研究预览功能

```
┌────────────────────────────────────────────────────────────┐
│              长期运行 Agent vs 普通 Agent                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  普通 Agent:                                               │
│  - 单次交互                                                │
│  - 中等复杂度任务                                          │
│  - 适合快速修改                                            │
│                                                            │
│  长期运行 Agent:                                           │
│  - 自主规划 → 执行 → 迭代                                  │
│  - 完成复杂的大规模重构                                    │
│  - 生成完整的 PR                                           │
│  - 适合需要多步骤的大型任务                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.3 多 Agent 协作系统

Cursor 背后的研究实现了**数千个 Agent 协同工作**：

```
┌─────────────────────────────────────────────────────────────┐
│                   多层 Agent 架构                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              Root Planner                           │   │
│  │         (理解整体目标，拆分任务)                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓ spawn                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              Sub-planners                           │   │
│  │         (负责子领域，完全拥有)                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓ spawn                           │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │
│  │   Worker 1   │  │   Worker 2   │  │   Worker N   │    │
│  │  (独立任务)   │  │  (独立任务)   │  │  (独立任务)   │    │
│  └──────────────┘  └──────────────┘  └──────────────┘    │
│                          ↓ handoff                         │
│                   返回结果给 Planner                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**性能数据：**
- 峰值：**~1000 commits/小时**
- 超过 **1000万次工具调用**
- 一周运行**无需人工干预**

## 4. 上下文管理

### 4.1 何时开新对话？

```
┌─────────────────────────────────────────────────────────────┐
│                     决策流程                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  开始新对话的情况:                                          │
│  ✓ 切换到不同的任务或功能                                   │
│  ✓ Agent 看起来困惑或重复犯同样错误                         │
│  ✓ 完成了某个逻辑单元的工作                                 │
│                                                             │
│  继续当前对话:                                              │
│  ✓ 迭代同一功能                                             │
│  ✓ 需要之前讨论的上下文                                     │
│  ✓ 调试 Agent 刚构建的东西                                  │
│                                                             │
│  ⚠️  长对话会导致 Agent 失去焦点                            │
│      (上下文累积噪音后效率下降)                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 @ 引用功能

| 引用方式 | 语法 | 用途 |
|----------|------|------|
| 文件 | `@filename.py` | 提供特定文件上下文 |
| 文件夹 | `@/src` | 提供整个目录 |
| 代码片段 | `@#函数名` | 引用特定代码块 |
| 过去对话 | `@Past Chats` | 引用历史对话 |
| 分支 | `@Branch` | 引用当前分支状态 |
| 文档 | `@Docs` | 引用外部文档 |

### 4.3 图像处理

Cursor Agent 可以直接处理图像：

```markdown
# 设计稿转代码
"实现这个设计稿的界面" + [粘贴设计图]

# 视觉调试
"为什么这个按钮样式不对" + [截图]
```

## 5. 扩展与自定义

### 5.1 Rules（规则）

**作用：** 提供项目级的持久上下文

```
.cursor/rules/
├── README.md          # 项目规范
├── coding-style.md    # 代码风格
└── workflow.md        # 工作流
```

**示例规则文件：**

```markdown
# .cursor/rules/project.md

# 命令
- npm run build: 构建项目
- npm run typecheck: 运行类型检查
- npm run test: 运行测试（优先单文件测试）

# 代码风格
- 使用 ES Modules (import/export)
- 禁止 CommonJS (require)
- 参考 components/Button.tsx 的组件结构

# 工作流
- 修改后始终运行 typecheck
- API 路由放在 app/api/ 目录
```

**最佳实践：**
- 保持规则简洁，引用文件而非复制内容
- 只在 Agent 重复犯同样错误时添加
- 将规则纳入版本控制

### 5.2 Skills（技能）

**作用：** 动态加载的专业能力

```
.cursor/
├── skills/
│   ├── tdd/
│   │   └── SKILL.md
│   └── code-review/
│       └── SKILL.md
└── hooks.json
```

**SKILL.md 示例：**

```markdown
# TDD Workflow

## 触发条件
当用户要求编写测试或进行 TDD 时

## 工作流程

1. 先写失败的测试
2. 运行测试确认失败
3. 编写最小实现代码
4. 运行测试确认通过
5. 重构

## 命令
- /tdd: 启动 TDD 流程
```

### 5.3 Hooks（钩子）

**作用：** 在特定时机执行自定义逻辑

```json
// .cursor/hooks.json
{
  "version": 1,
  "hooks": {
    "stop": [{ "command": "bun run .cursor/hooks/grind.ts" }]
  }
}
```

**应用场景：**
- 测试失败时自动重试
- 提交前检查代码规范
- 自动化部署

### 5.4 MCP (Model Context Protocol)

**作用：** 连接外部工具和服务

```
┌─────────────────────────────────────────────────────────────┐
│                     MCP 集成                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │  Slack  │  │ Datadog │  │  Sentry │  │ Figma   │      │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘      │
│       │            │            │            │            │
│       └────────────┴────────────┴────────────┘            │
│                          ↓                                  │
│                   MCP Server                               │
│                          ↓                                  │
│                   Cursor Agent                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**常用 MCP 服务器：**
- **Figma**: 设计稿直接转代码
- **Slack**: 读取消息、发送通知
- **Datadog**: 查询日志
- **Sentry**: 调试错误
- **数据库**: 直接查询

## 6. 最佳实践

### 6.1 测试驱动开发 (TDD)

```markdown
# 正确流程：

1. "为这个函数写测试，测试 [输入] → [预期输出]"
   → 不要创建不存在的功能

2. "运行测试，确认失败"
   → 此时不要写实现代码

3. 提交测试

4. "写代码让测试通过，不要修改测试"
   → 迭代直到全部通过

5. 提交实现
```

### 6.2 代码审查

```markdown
# 审查层次：

1. 生成中
   - 按 Esc 打断偏离方向的修改

2. 生成后
   - 点击 Review → Find Issues
   - Agent 逐行分析问题

3. PR 审查
   - Bugbot 自动审查 PR
   - 支持自动修复 (Autofix)
```

### 6.3 Git 工作流

创建常用命令：

```markdown
# .cursor/commands/pr.md

# /pr - 创建 Pull Request

1. 查看 staged 和 unstaged 更改
2. 写清晰的 commit message
3. commit 并 push 到当前分支
4. 使用 gh pr create 创建 PR
5. 返回 PR URL
```

**常用命令示例：**
- `/fix-issue [number]`: 获取 issue → 定位代码 → 修复 → 开 PR
- `/review`: 运行 linter → 检查问题 → 总结
- `/update-deps`: 检查依赖更新 → 逐个更新 → 运行测试

### 7.2 Cloud Agents

**最新功能：** 云端运行的 Agent

```
┌─────────────────────────────────────────────────────────────┐
│                  Cloud vs Local Agent                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Local Agent:                                               │
│  - 在你的机器上运行                                         │
│  - 访问本地代码库                                          │
│  - 需要你授权操作                                           │
│                                                             │
│  Cloud Agent:                                               │
│  - 在隔离的 VM 中运行                                       │
│  - 自己配置代码库                                           │
│  - 生成视频/截图/日志作为产物                               │
│  - 可以从 Slack/GitHub 触发                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 性能优化

```markdown
# 提升 Agent 效率的技巧：

1. 精确的上下文
   - 知道具体文件？直接 @文件名
   - 不知道？让 Agent 自己搜索

2. 适时开新对话
   - 效率下降时立即新建

3. 使用计划模式
   - 大任务先用 Plan Mode

4. 避免无关文件
   - 包含无关文件会混淆 Agent
```
